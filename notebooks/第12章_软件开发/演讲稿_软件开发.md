### 一、引入——软件不是"写完就完" (5 分钟)

同学们好。前几章我们学了算法设计、数据结构和编程技术。你们可能觉得会写代码就能做软件了。但现实中，写代码只是软件开发中**很小的一部分**。

给大家说一个数据：一个典型的商业软件项目中，**写代码只占总工作量的 20%**。剩下的 80% 是需求分析、设计、测试和维护。

为什么？因为做软件跟盖房子一样——你不可能拿到砖就开始砌，你需要一个完整的流程来确保最终成品符合要求、质量可靠、后期可维护。

今天我们就来学这个完整的流程——**程序开发生命周期 (Program Development Life Cycle, PDLC)**。

---

### 二、五个阶段 (15 分钟)

#### 2.1 概述

【板书】程序开发生命周期的五个阶段：

```
分析 → 设计 → 编码 → 测试 → 维护
Analysis → Design → Coding → Testing → Maintenance
```

我们用盖房子来类比每一个阶段。

#### 2.2 分析 (Analysis)

盖房子之前，你要先跟客户聊：你要几间卧室？有没有停车位需求？预算多少？

软件开发也一样。分析阶段要回答的核心问题是：**这个软件到底要做什么？**

【板书】分析阶段的任务：
- 收集需求 (Requirements gathering)——跟用户访谈、发问卷、观察工作流程
- 可行性研究 (Feasibility study)——技术上能不能做？经济上值不值得？
- 产出需求说明书 (Requirements specification)

分析阶段最容易犯的错误是**自以为理解了用户需求**。很多项目失败不是因为代码写得差，而是因为做出来的东西不是用户想要的。

#### 2.3 设计 (Design)

分析清楚需求后，建筑师要画图纸。软件设计就是"画图纸"：

【板书】设计阶段的产出：
- 结构图 (Structure charts)——模块之间的关系
- 数据结构设计——用什么数组、记录、文件
- 算法设计——伪代码、流程图
- 界面设计——用户看到什么
- 测试计划——怎么验证软件正确

我们下节课会详细学设计阶段的工具。

#### 2.4 编码 (Coding)

这是大家最熟悉的阶段——写代码。把设计翻译成某种编程语言。

但注意，好的编码不只是"能运行"。还要：
- 代码可读（有意义的变量名、注释）
- 遵循编码规范
- 模块化（用过程和函数）

#### 2.5 测试 (Testing)

盖完房子要验收，软件写完要测试。测试的目的是**发现错误 (bugs)**。

【提问】为什么不能跳过测试阶段？反正我自己写的代码我觉得没问题啊？

……因为人类天生不擅长发现自己的错误。就像你写作文很难发现自己的错别字一样。更重要的是，即使每一行代码单独都是对的，组合在一起可能会出问题。我们第三节课会详细讲测试。

#### 2.6 维护 (Maintenance)

房子交付后还需要维护——水管堵了要修、想加一个房间要改建、法规变了要合规。

软件也一样。维护分三种：
- **纠正性维护 (Corrective)**：修 bug
- **适应性维护 (Adaptive)**：适配新环境（如新操作系统）
- **完善性维护 (Perfective)**：添加新功能、提升性能

维护是成本最高的阶段——一个软件生命周期中，维护占总成本的 60-80%！

### 三、开发模型 (20 分钟)

五个阶段定下来了，但按什么**顺序**执行呢？不同的项目适合不同的方式。这就是**开发模型 (Development Models)**。

#### 3.1 瀑布模型 (Waterfall Model)

【板书】

```
分析
  ↓
设计
  ↓
编码
  ↓
测试
  ↓
维护
```

像瀑布一样从上到下，一步步走，**每个阶段完成后才进入下一个**。不回头。

**优点**：
- 简单，容易理解和管理
- 每个阶段有明确的文档产出
- 适合需求非常清楚、不太会变的项目

**缺点**：
- 不灵活！如果在测试阶段发现需求理解错了，回头改的代价巨大
- 用户要等到最后才能看到成品
- 真实世界中需求很少是一开始就完全确定的

**适用场景**：政府系统、航空航天、医疗设备——需求明确、安全要求高、不能频繁修改。

#### 3.2 迭代模型 (Iterative Model)

【板书】

```
    分析 → 设计 → 编码 → 测试
    ↑                        ↓
    ← ← ← 评估反馈 ← ← ← ←
```

把项目分成多个迭代 (iteration)，每次迭代都经过分析-设计-编码-测试的完整循环。每次迭代交付一个**可运行的部分版本**。

打个比方：瀑布模型是一次性盖好整栋楼再交付；迭代模型是先交付一层能住的，然后再加第二层、第三层，每次加一层都是完整可用的。

**优点**：
- 用户早期就能看到部分成品，及时反馈
- 每次迭代可以调整需求，灵活
- 风险更低——问题早发现早解决

**缺点**：
- 管理更复杂——需要频繁的沟通和规划
- 如果架构设计不好，后期迭代可能要大改

**适用场景**：需求不完全确定、需要用户频繁反馈的项目。现代大部分软件开发都用迭代模型的某种变体。

#### 3.3 RAD 模型 (Rapid Application Development)

【板书】RAD = 快速应用开发

RAD 的核心思想是**快速做出原型 (Prototype)**，让用户看到、摸到，然后根据反馈快速修改。

流程：
```
需求 → 快速原型 → 用户评估 → 修改 → ... → 最终产品
```

它强调：
- **速度**——快速出原型
- **用户参与**——用户全程参与反馈
- **可视化工具**——用拖拽等工具快速搭建界面

**优点**：
- 速度快，用户满意度高
- 需求变化的代价低
- 适合界面密集的应用（如手机 App、网站）

**缺点**：
- 不适合大型复杂系统
- 原型可能变成最终产品（质量不够好）
- 需要经验丰富的开发者

**适用场景**：小型项目、时间紧迫、用户界面为主的应用。

#### 3.4 模型对比

【板书】

| 特性 | 瀑布 | 迭代 | RAD |
|------|------|------|-----|
| 灵活性 | 低 | 中 | 高 |
| 用户参与 | 少 | 中 | 多 |
| 文档量 | 多 | 中 | 少 |
| 适合项目 | 大型稳定需求 | 中大型 | 小型快速 |
| 风险 | 高（问题晚发现） | 中 | 低 |

【提问】如果你要开发一个银行核心交易系统，你会选哪个模型？如果是一个校园活动报名的小程序呢？

……银行系统选瀑布或迭代——需求明确、安全要求高。小程序选 RAD——快速出原型、用户频繁反馈。

### 四、真实案例分享 (8 分钟)

给大家讲两个真实的软件项目故事。

**案例 1：英国 NHS 电子病历系统 (NPfIT)**

2002 年英国政府启动了一个 120 亿英镑的项目，要给全国的医院做统一的电子病历系统。用的是瀑布模型。结果呢？2011 年宣布项目失败，花了 120 亿英镑只交付了很小一部分。

为什么失败？因为各医院的需求差异巨大，而瀑布模型不允许灵活调整。几年后需求变了，系统还在按最初的需求开发。这是选错开发模型的经典反面教材。

**案例 2：微信的迭代开发**

微信 2011 年 1.0 版本发布时只有最基本的聊天功能。然后不断迭代：加了语音消息、朋友圈、支付、小程序……每次迭代都根据用户反馈决定下一步做什么。这就是迭代模型的典范。如果微信一开始就想做一个"万能 App"，可能到现在还没发布。

### 一、设计阶段在做什么？(5 分钟)

同学们，上节课讲了开发生命周期的五个阶段。今天我们深入第二个阶段——**设计**。

如果分析阶段回答的是"做什么"，设计阶段回答的就是"怎么做"。具体来说，我们要用**结构图 (Structure Charts)** 来规划程序的模块结构，用**状态转换图 (State-Transition Diagrams)** 来描述系统的行为，然后把设计转化为伪代码。

---

### 二、结构图 (Structure Charts) (18 分钟)

#### 2.1 什么是结构图？

结构图是展示程序**模块之间层次关系**的图表。它回答的问题是：这个程序由哪些模块组成？模块之间如何调用？数据怎么传递？

【板书】结构图的基本元素：

| 符号 | 含义 |
|------|------|
| 矩形 | 模块（过程或函数） |
| 连接线 | 调用关系（上面调用下面） |
| 带箭头的小圆圈 | 数据传递（空心=数据，实心=控制标志） |
| 菱形 | 条件调用 |
| 弧形箭头 | 循环调用 |

#### 2.2 ATM 结构图示例

让我们设计一个简单 ATM 机的结构图。

【板书】（画结构图）

```
                    ATM 系统
          ┌──────────┼──────────┐
          │          │          │
      验证用户    处理交易    打印凭条
          │      ┌───┼───┐
      检查PIN   取款  存款  查询余额
                 │
            更新余额
```

最顶层是主模块"ATM 系统"。它调用三个子模块："验证用户"、"处理交易"、"打印凭条"。其中"处理交易"又有三个子模块："取款"、"存款"、"查询余额"。

这就是自顶向下的分解。还记得第 9 章学的分解 (Decomposition) 吗？结构图就是分解结果的可视化表示。

#### 2.3 数据传递

模块之间怎么传数据？在结构图上用**箭头和小圆圈**表示。

【板书】

```
     ATM 系统
        │
    ↓PIN  ↑Valid?
        │
    验证用户
```

- `↓PIN`：主模块把 PIN 传给验证模块（向下传数据）
- `↑Valid?`：验证模块把结果（是否有效）传回主模块（向上返回控制标志）

这不就是我们第 11 章学的参数传递吗？结构图上的数据箭头对应伪代码中的参数，返回箭头对应 RETURN 值。

#### 2.4 从结构图到伪代码

有了结构图，转换成伪代码就很自然了：

【板书】

```
PROCEDURE ATMSystem()
  IF VerifyUser()
    THEN
      CALL ProcessTransaction()
      CALL PrintReceipt()
  ELSE
    OUTPUT "Invalid PIN"
  ENDIF
ENDPROCEDURE

FUNCTION VerifyUser() RETURNS BOOLEAN
  DECLARE PIN : STRING
  OUTPUT "Enter PIN: "
  INPUT PIN
  IF PIN = StoredPIN
    THEN RETURN TRUE
    ELSE RETURN FALSE
  ENDIF
ENDFUNCTION

PROCEDURE ProcessTransaction()
  OUTPUT "1. Withdraw  2. Deposit  3. Balance"
  INPUT Choice
  CASE OF Choice
    1 : CALL Withdraw()
    2 : CALL Deposit()
    3 : CALL CheckBalance()
  ENDCASE
ENDPROCEDURE
```

【演示】运行 Notebook 中 ATM 结构图的演示

【提问】如果用户输入错误的 PIN，我们想让他最多尝试 3 次，应该怎么修改？

……在 VerifyUser 里加一个 FOR 循环或 REPEAT 循环，用计数器记录尝试次数。这就是逐步细化——先画大的结构，再逐个细化每个模块的内部逻辑。

### 三、状态转换图 (State-Transition Diagrams) (17 分钟)

#### 3.1 什么是状态转换图？

有些系统的行为取决于它当前处于什么**状态 (state)**。比如一个交通灯，它不是随时都能从红灯变成绿灯的——必须先红灯，然后红黄灯（在英国），然后绿灯，然后黄灯，再回到红灯。

状态转换图就是描述**系统在不同状态之间如何转换**的图表。

【板书】状态转换图的元素：
- **状态 (State)**：系统在某一时刻的状况，用圆角矩形表示
- **转换 (Transition)**：从一个状态到另一个状态的变化，用箭头表示
- **事件/条件 (Event/Condition)**：触发转换的原因，标注在箭头上
- **起始状态**：实心圆点
- **终止状态**：圈中有实心圆

#### 3.2 交通灯示例

【板书】

```
 ●→ [红灯] ──定时器到期──→ [红黄灯] ──定时器到期──→ [绿灯]
       ↑                                                │
       │                                          定时器到期
       │                                                ↓
       └──────────定时器到期──────────── [黄灯] ←──────┘
```

四个状态：红灯、红黄灯、绿灯、黄灯。
转换条件：定时器到期。
注意：这是一个**循环**——从黄灯回到红灯，永远循环下去。

这就是英国交通灯的实际工作方式。在中国没有"红黄灯"阶段，流程会稍有不同。

#### 3.3 更复杂的例子——门禁系统

一个刷卡门禁：

【板书】

```
 ●→ [锁定] ──刷有效卡──→ [解锁]
       ↑    ←─刷无效卡─┘    │
       │                 超时 / 通过
       └──────────────────┘
```

状态：锁定、解锁
转换：
- 锁定 + 刷有效卡 → 解锁
- 锁定 + 刷无效卡 → 锁定（不变，报警）
- 解锁 + 超时或有人通过 → 锁定

#### 3.4 状态转换表

状态转换图也可以用**表格**表示：

【板书】

| 当前状态 | 事件 | 下一状态 | 动作 |
|---------|------|---------|------|
| 锁定 | 刷有效卡 | 解锁 | 开门，亮绿灯 |
| 锁定 | 刷无效卡 | 锁定 | 报警，亮红灯 |
| 解锁 | 超时 | 锁定 | 关门 |
| 解锁 | 有人通过 | 锁定 | 关门 |

表和图表达的信息完全一样，只是表现形式不同。考试中可能要你画图或填表。

#### 3.5 从状态转换图到伪代码

【板书】

```
State ← "LOCKED"
WHILE TRUE DO
  CASE OF State
    "LOCKED":
      Event ← WaitForEvent()
      IF Event = "ValidCard"
        THEN
          CALL OpenDoor()
          State ← "UNLOCKED"
        ELSE
          CALL SoundAlarm()
      ENDIF
    "UNLOCKED":
      Event ← WaitForEvent()
      IF Event = "Timeout" OR Event = "PersonPassed"
        THEN
          CALL CloseDoor()
          State ← "LOCKED"
      ENDIF
  ENDCASE
ENDWHILE
```

【演示】运行 Notebook 中状态转换的交互式演示

【提问】如果门禁系统要增加一个功能：连续 3 次刷无效卡就报警锁定 5 分钟，状态转换图会怎么变？

……需要增加一个"临时锁定"状态，并且在"锁定"状态中增加一个失败计数器。这就是设计的价值——在画图的时候发现问题比写完代码后发现问题要便宜得多。

### 四、设计到实现的完整流程 (8 分钟)

现在我们来看一个完整的设计流程：

**问题**：设计一个简单的计算器，支持加减乘除。

**步骤 1：结构图**

```
           计算器
    ┌────┬────┬────┐
   加法  减法  乘法  除法
```

**步骤 2：识别数据流**
- 用户输入两个数和运算符
- 每个模块接收两个数，返回结果

**步骤 3：伪代码**

```
FUNCTION Add(a : REAL, b : REAL) RETURNS REAL
  RETURN a + b
ENDFUNCTION

FUNCTION Subtract(a : REAL, b : REAL) RETURNS REAL
  RETURN a - b
ENDFUNCTION

FUNCTION Multiply(a : REAL, b : REAL) RETURNS REAL
  RETURN a * b
ENDFUNCTION

FUNCTION Divide(a : REAL, b : REAL) RETURNS REAL
  IF b = 0
    THEN
      OUTPUT "Error: Division by zero!"
      RETURN 0
  ELSE
    RETURN a / b
  ENDIF
ENDFUNCTION

// 主程序
OUTPUT "Enter first number: "
INPUT Num1
OUTPUT "Enter operator (+, -, *, /): "
INPUT Op
OUTPUT "Enter second number: "
INPUT Num2

CASE OF Op
  "+" : Result ← Add(Num1, Num2)
  "-" : Result ← Subtract(Num1, Num2)
  "*" : Result ← Multiply(Num1, Num2)
  "/" : Result ← Divide(Num1, Num2)
  OTHERWISE : OUTPUT "Invalid operator"
ENDCASE
OUTPUT "Result = ", Result
```

注意 Divide 函数中的除零检查——这就是设计阶段思考边界情况的好处。

### 一、为什么测试如此重要？(8 分钟)

同学们，今天是第 12 章的最后一节课。我们聊一个很多初学者不太重视但极其关键的话题——**测试 (Testing)**。

先给大家讲两个真实的灾难性故事。

**故事一：Therac-25 放射治疗机（1985-1987）**

Therac-25 是一台用于癌症治疗的放射治疗机。由于软件中的一个竞态条件 (race condition) bug，这台机器在两年间给至少 6 名患者施加了致命剂量的辐射。有的患者因此死亡。

问题出在哪？之前的版本有硬件安全锁，但 Therac-25 为了省钱去掉了硬件安全锁，完全依赖软件来控制剂量。而软件没有经过充分测试。

**故事二：Ariane 5 火箭爆炸（1996）**

这个故事我们在第 10 章讲数据类型时提到过。64 位浮点数转 16 位整数时溢出，火箭发射 37 秒后自毁。3.7 亿美元没了。

讽刺的是，这段代码是从 Ariane 4 直接复制过来的。在 Ariane 4 上运行没问题，因为 Ariane 4 的飞行参数不会超过 16 位整数的范围。但 Ariane 5 更快了，参数范围变了，而**没有人重新测试这段代码**。

荷兰计算机科学家 Edsger Dijkstra 说过一句名言：

【板书】"Testing can show the presence of bugs, but never their absence."
— Dijkstra

翻译：测试只能证明 bug 的存在，但永远无法证明 bug 不存在。

这意味着测试不能保证你的程序完全正确，但不测试就更加不行。测试是在不完美的世界里保护用户安全的最后一道防线。

### 二、三种错误类型 (10 分钟)

编程中的错误（bug）分为三类。

#### 2.1 语法错误 (Syntax Error)

【板书】语法错误 = 违反了编程语言的语法规则

就像你写英语作文把 "I am" 写成了 "I is"——语法不对。

```python
# 语法错误示例
print("Hello"      # 缺少右括号
for i in range(10) # 缺少冒号
x = = 5            # 多了一个等号
```

好消息是：语法错误**最容易发现**。编译器/解释器会直接报错，告诉你哪一行有问题。程序根本运行不起来。

#### 2.2 逻辑错误 (Logic Error)

【板书】逻辑错误 = 程序能运行，但结果不对

这是最危险的错误！程序看起来一切正常，不报错，不崩溃，但给出的答案是错的。

```python
# 逻辑错误示例
average = num1 + num2 + num3 / 3  # 应该是 (num1 + num2 + num3) / 3
# 缺少括号，导致只有 num3 除以 3
```

【提问】为什么逻辑错误最危险？

……因为你可能根本不知道它存在！程序跑着跑着，给你一个看起来合理但其实错误的结果。你还以为是对的。Ariane 5 就是逻辑错误——程序运行了，但溢出的逻辑处理是错误的。

#### 2.3 运行时错误 (Runtime Error)

【板书】运行时错误 = 程序运行过程中崩溃

语法没问题，逻辑也"看起来"对，但运行到某个点就崩溃了。

```python
# 运行时错误示例
x = 10 / 0            # 除以零
names = ["Alice", "Bob"]
print(names[5])        # 索引越界
num = int("hello")     # 类型转换失败
```

运行时错误通常由**异常情况**引起——你没预料到的输入或状态。好的程序会做**错误处理**来防止崩溃。

【板书】三种错误对比：

| 类型 | 发现难度 | 程序行为 | 类比 |
|------|---------|---------|------|
| 语法错误 | 最容易 | 无法运行 | 写了错别字 |
| 逻辑错误 | 最难 | 运行但结果错 | 算错了账 |
| 运行时错误 | 中等 | 运行中崩溃 | 走路摔跤 |

### 三、测试方法 (15 分钟)

#### 3.1 白盒测试 (White-box Testing)

【板书】白盒测试 = 看着代码测试，确保每条路径都被执行到

你打开"白色的盒子"，看到里面的代码结构，然后设计测试用例来覆盖每一条路径。

```python
def classify(score):
    if score >= 90:
        return "A"
    elif score >= 80:
        return "B"
    else:
        return "C"
```

白盒测试需要测试：score >= 90 的路径、80 <= score < 90 的路径、score < 80 的路径。每条路径至少执行一次。

#### 3.2 黑盒测试 (Black-box Testing)

【板书】黑盒测试 = 不看代码，只根据输入输出规格来测试

你不知道（或假装不知道）程序内部怎么实现的，只关心：给了这个输入，应该得到什么输出？

黑盒测试通常由非开发人员（如测试工程师）来做，因为他们没有"我知道代码怎么写的所以我觉得它是对的"这种偏见。

#### 3.3 Alpha 和 Beta 测试

【板书】

- **Alpha 测试**：在开发环境内部进行，由开发团队或内部测试人员完成
- **Beta 测试**：把软件发给一部分真实用户试用，收集反馈

你们平时玩游戏的"Beta 版"就是这个意思——游戏还没正式发布，先给一部分玩家试玩找 bug。

#### 3.4 验收测试 (Acceptance Testing)

由最终用户或客户进行，确认软件满足需求。通过验收测试，项目才能算交付成功。

### 四、测试数据的选择 (10 分钟)

知道用什么方法测试了，但测试的时候输入什么数据呢？不能随便乱输——要**有策略地**选择测试数据。

#### 4.1 三种测试数据

【板书】

假设程序要求输入一个 1-100 之间的整数：

| 类型 | 英文 | 含义 | 示例 |
|------|------|------|------|
| 正常数据 | Normal | 合理的、预期内的输入 | 50, 75, 1, 100 |
| 异常数据 | Abnormal/Erroneous | 不合理的、不应被接受的输入 | -5, 150, "abc", 3.14 |
| 边界数据 | Boundary/Extreme | 在合法范围边缘的输入 | 0, 1, 100, 101 |

#### 4.2 为什么边界数据特别重要？

大部分 bug 出现在边界上！

【提问】如果条件写成了 `IF Score > 0 AND Score < 100`，哪些边界值会暴露这个 bug？

……输入 0 和 100。按照需求 1-100 应该是合法的，但这个条件把 0 排除了（>0 不包含 0）...等等，0 本来就不在 1-100 范围内。但 100 呢？`Score < 100` 把 100 排除了！应该是 `Score <= 100`。

你看，输入 50 的话程序看起来完全正常，只有输入 100 这个边界值才能发现 bug。这就是边界测试的价值。

#### 4.3 设计测试用例表

【板书】

| 测试编号 | 输入数据 | 数据类型 | 预期输出 | 实际输出 | 通过？ |
|---------|---------|---------|---------|---------|--------|
| 1 | 50 | Normal | Pass | | |
| 2 | 85 | Normal | Pass | | |
| 3 | 30 | Normal | Fail | | |
| 4 | -5 | Abnormal | Error msg | | |
| 5 | "abc" | Abnormal | Error msg | | |
| 6 | 0 | Boundary | Error msg | | |
| 7 | 1 | Boundary | Fail | | |
| 8 | 100 | Boundary | Pass | | |
| 9 | 101 | Boundary | Error msg | | |

考试中经常要求你设计测试用例表。关键是：三种类型的数据都要有，特别是边界数据。

### 五、维护类型 (5 分钟)

软件发布后，工作还没结束。维护是软件生命周期中持续时间最长、成本最高的阶段。

【板书】三种维护类型：

| 类型 | 英文 | 触发原因 | 类比 |
|------|------|---------|------|
| 纠正性维护 | Corrective | 发现了 bug | 修漏水的水管 |
| 适应性维护 | Adaptive | 环境变化 | 把燃气灶换成电磁炉 |
| 完善性维护 | Perfective | 用户需要新功能 | 加建一个阳台 |

**纠正性维护 (Corrective)**：修 bug。用户报告了一个错误，你要找到原因并修复。

**适应性维护 (Adaptive)**：适应新环境。比如 Windows 升级了，你的软件要兼容新版本；法律法规变了，系统要做相应调整。

**完善性维护 (Perfective)**：添加新功能或提升性能。用户说"你这个软件挺好用，能不能再加个 XX 功能？"这就是完善性维护。

【提问】微信每次更新（新版本），属于哪种维护？

……通常是完善性维护（加新功能）和纠正性维护（修 bug）的结合。每次更新日志里的"修复已知问题"就是纠正性维护，"新增 XX 功能"就是完善性维护。
