## 开场：为什么需要操作系统？（约 5 分钟）

同学们好！今天我们开始学习第五章——系统软件。我想先问大家一个问题。

【提问】你们每天打开电脑、打开手机，第一个看到的东西是什么？

对，是桌面——Windows 桌面、macOS 桌面、或者手机的主屏幕。这些东西是谁提供的？是操作系统。

但是你们有没有想过：如果没有操作系统，计算机会怎样？

在 1940 到 1950 年代，早期的计算机确实没有操作系统。那时候程序员要怎么写程序？他们必须直接操作硬件——用物理开关、打孔卡片来输入每一条指令。想让打印机打印东西？你得知道打印机的物理地址，手动发送电信号。想用内存？你得自己计算哪些内存地址是空闲的，自己管理分配。

这就像什么呢？想象你去一家没有服务员的餐厅——你得自己去厨房找食材，自己开灶台，自己烹饪，吃完还要自己洗锅。你去餐厅是为了吃饭，不是为了干这些杂活。

操作系统就是这家餐厅的服务员和厨师团队。你只需要告诉它"我要什么"，它帮你处理一切细节。

【板书】
```
操作系统 (Operating System, OS)
= 管理计算机硬件资源、为应用程序提供服务的系统软件
= 用户与硬件之间的"桥梁"
```

【演示】打开 Notebook 01，运行 Cell 3——操作系统的分层结构图。

大家看这张图。操作系统就像一个洋葱，最内层是硬件（CPU、内存、硬盘），外面包了一层操作系统内核（Kernel），再外面是各种系统服务，最外层才是用户和应用程序。每一层只和相邻的层打交道，这就是"分层抽象"的思想。

好，那操作系统具体做哪些事呢？我们可以归纳为**五大管理任务**。

## 第一节：内存管理（约 8 分钟）

### 为什么需要内存管理？

【提问】同学们，假设你的电脑有 8GB 内存，你同时打开了浏览器、Word、音乐播放器、微信。谁来决定每个程序分到多少内存？

这就是操作系统的**内存管理（Memory Management）**功能。

把 RAM 想象成一个**大办公桌**。办公桌大小是固定的——比如 8GB。你要同时做好几件事——写作业、画画、看参考书——每件事都需要占据桌面的一部分。如果你把所有东西都摊在桌上，桌面不够用了怎么办？你就得把暂时不用的东西放到抽屉里（这就是虚拟内存，用硬盘来模拟 RAM）。

操作系统在内存管理方面要做三件事：

【板书】
```
内存管理的三大任务：
1. 分配 (Allocation)：给每个程序分配一块内存空间
2. 保护 (Protection)：确保程序A不会跑到程序B的内存区域搞破坏
3. 回收 (Deallocation)：程序关闭后，释放它占用的内存
```

【演示】运行 Cell 5——三种内存分配方式的可视化。

大家看这三种方式。第一种是**分区分配（Partitioning）**，就像把办公桌划成几个固定区域，每个程序放一个区域。问题是区域大小固定，如果一个小程序放在一个大区域里，剩余空间就浪费了。

第二种是**分页（Paging）**，把内存切成大小相同的小块（叫"页"），程序需要多少就分多少页。这样碎片问题就小多了。

第三种是**虚拟内存（Virtual Memory）**，当 RAM 不够时，操作系统把暂时不用的数据临时挪到硬盘上。这就是为什么你的电脑有 8GB RAM 却能同时运行占用 12GB 内存的程序——虽然会变慢（因为硬盘比 RAM 慢得多），但至少不会崩溃。

## 第二节：文件管理（约 7 分钟）

硬盘就像一个**巨大的仓库**，里面有成千上万个货架（扇区）。如果没有一个管理系统，你存进去的东西根本找不回来。

操作系统的**文件管理（File Management）**就是这个仓库的管理员。它负责：

【板书】
```
文件管理的核心功能：
1. 命名：给文件取名字（包括扩展名 .docx .jpg .py）
2. 组织：用目录/文件夹的树形结构来组织文件
3. 存取：知道文件的物理位置，帮你读写数据
4. 权限：控制谁能看、谁能改、谁能删
```

【提问】你们平时在电脑上是怎么整理文件的？是全部扔在桌面上，还是会建文件夹分类？

其实你们建文件夹、重命名、复制粘贴——这些操作背后都是操作系统的文件管理在工作。你看到的"文件夹"只是操作系统给你的一个友好界面，硬盘上其实没有"文件夹"这个物理概念，只有一堆二进制数据块。操作系统维护了一个目录表，记录了每个文件名对应哪些数据块。

【演示】运行 Cell 9——模拟简单文件系统的代码。

大家看这个模拟文件系统。我们可以创建目录、创建文件、列出目录内容。这就是操作系统在幕后做的事情。

关于**文件权限（File Permissions）**，在 Linux 系统里，每个文件有三组权限——所有者（Owner）、所属组（Group）、其他人（Others）。每组可以设置读（r）、写（w）、执行（x）三种权限。这种精细的权限控制是多用户系统安全的基础。

## 第三节：安全管理（约 5 分钟）

想象一栋办公楼——不是每个人都能进入每个房间。访客需要在前台登记（认证），不同员工有不同的门禁卡权限（授权），保安会巡逻防止可疑人员（监控）。

操作系统的安全管理也是类似的分层结构：

【板书】
```
安全管理三个层面：
1. 认证 (Authentication)：验证"你是谁"——用户名+密码
2. 授权 (Authorization)：确定"你能做什么"——权限等级
3. 审计 (Auditing)：记录"你做了什么"——日志系统
```

【演示】运行 Cell 11——简单登录系统的模拟。

大家注意看，这段代码展示了一个重要概念：密码不是以明文存储的，而是存储密码的**哈希值（Hash）**。即使有人偷看了数据库，看到的也是一串无意义的字符，无法反推出原始密码。这就是为什么你忘记密码时，网站只能让你"重置密码"而不能"找回密码"——因为它自己也不知道你的原始密码是什么。

## 第四节：硬件管理与设备驱动（约 5 分钟）

【提问】你们有没有遇到过这种情况：买了一个新的打印机或游戏手柄，插上电脑后不能用，要先"安装驱动程序"？

世界上有成千上万种不同的硬件设备——不同品牌的打印机、鼠标、显卡、声卡。操作系统不可能提前知道每种设备怎么工作。所以它采用了一个聪明的策略：**设备驱动程序（Device Driver）**。

【演示】运行 Cell 13——设备驱动架构图。

大家看这张图。操作系统提供一个统一的接口——不管你用什么牌子的打印机，应用程序只需要调用"打印"这个标准指令。然后设备驱动程序负责把这个标准指令翻译成特定打印机能听懂的具体命令。

这就像翻译的工作。你说中文，打印机说"佳能语"或"惠普语"。设备驱动程序就是中间的翻译官。

操作系统还管理**输入输出（I/O）**——数据从键盘进来、到屏幕出去、到打印机出去，这些数据流都需要协调管理，避免冲突。比如两个程序同时想用打印机，操作系统会用**打印队列（Print Queue）**来排队，一个一个来。

## 第五节：进程管理与时间片轮转（约 8 分钟）

这是操作系统最"魔法"的部分。

【提问】你的 CPU 通常只有几个核心——比如 4 核或 8 核。但你能同时运行几十个甚至上百个程序。这怎么做到的？

答案是：CPU 并不是真的"同时"运行所有程序。它在不同程序之间**快速切换**，每个程序轮流执行一小段时间（通常只有几十毫秒）。因为切换速度极快，你感觉所有程序都在"同时"运行。

这个机制叫做**时间片轮转（Time Slicing / Round Robin）**。

首先我们要区分两个概念：

【板书】
```
程序 (Program) = 存在硬盘上的代码文件（静态的，像一本食谱）
进程 (Process) = 正在运行的程序实例（动态的，像正在做的一道菜）

一个程序可以同时有多个进程（比如你打开两个 Chrome 窗口）
```

【演示】运行 Cell 15——时间片轮转的可视化。

大家看这个可视化。横轴是时间，每一行代表一个进程。CPU 像一个旋转木马一样，轮流给每个进程分配一小段时间（时间片）。进程 A 运行 30 毫秒，暂停；CPU 切换到进程 B，运行 30 毫秒，暂停；再切换到进程 C……一圈下来再回到进程 A。

每次切换时，操作系统需要保存当前进程的状态（寄存器值、程序计数器等），加载下一个进程的状态。这叫做**上下文切换（Context Switching）**。它有一定开销，所以时间片不能太短（切换太频繁会浪费时间），也不能太长（用户会感觉程序"卡住了"）。

除了五大管理功能，操作系统还附带了一些**实用工具软件（Utility Software）**。这些小工具帮助你维护和优化系统。

【板书】
```
常见实用工具：
1. 磁盘碎片整理 (Disk Defragmentation)
2. 磁盘格式化 (Disk Formatting)
3. 文件压缩 (File Compression)
4. 备份工具 (Backup)
5. 杀毒软件 (Anti-virus) —— 有时是第三方
```

【演示】运行 Cell 17——碎片整理前后的可视化对比。

大家看这个对比。左边是碎片化的硬盘——一个文件的数据块散落在硬盘的各个角落。读取这个文件时，磁头要到处跳来跳去，速度很慢。右边是碎片整理之后——同一个文件的数据块被排列在一起，磁头可以顺序读取，速度快很多。

这就像整理房间。你的课本散落在桌上、床上、地上，找起来很费劲。把它们整理到一起放在书架上，找起来就快多了。

注意：**SSD（固态硬盘）不需要碎片整理**！因为 SSD 没有物理磁头，访问任何位置的速度都一样。对 SSD 做碎片整理反而会缩短它的寿命。

## 第七节：程序库（约 3 分钟）

最后我们来说**程序库（Program Library）**。

【提问】你们写 Python 程序的时候，有没有用过 `import math` 或者 `import random`？

这些就是程序库！程序库是一组预先编写好的、可重复使用的代码模块。你不需要从零开始造轮子——别人已经帮你写好了数学运算、随机数生成、文件处理等各种功能。

【演示】运行 Cell 21——展示使用程序库的威力。

程序库有两种形式：
- **静态链接库（Static Library）**：编译时把库的代码直接复制进你的程序。优点是独立运行，缺点是文件体积大。
- **动态链接库（Dynamic Link Library, DLL）**：运行时才加载库。多个程序可以共享同一份库文件，节省内存和硬盘空间。Windows 里的 .dll 文件、Linux 里的 .so 文件就是动态链接库。

你们可能见过 Windows 报错"缺少 xxx.dll 文件"——这就是因为程序找不到它依赖的动态链接库。

## 第一讲总结（约 2 分钟）

我们来回顾一下操作系统的全景图：

【板书】
```
操作系统 (OS)
├── 内存管理：分配、保护、回收 RAM
├── 文件管理：命名、组织、存取、权限
├── 安全管理：认证、授权、审计
├── 硬件管理：设备驱动、I/O 协调
├── 进程管理：时间片轮转、上下文切换
├── 实用工具：碎片整理、备份、压缩、杀毒
└── 程序库：静态库、动态链接库
```

记住核心思想：操作系统是一个**资源管理器（Resource Manager）**和**抽象层（Abstraction Layer）**。它管理硬件资源，并为应用程序提供简洁的接口，让程序员不用关心底层细节。

**（对应 Notebook：02_语言翻译器.ipynb）**

## 开场：翻译的困境（约 5 分钟）

上节课我们讲了操作系统。今天我们来讲系统软件的另一大类——**语言翻译器（Language Translators）**。

【提问】同学们，你们现在写程序用的是什么语言？Python？Java？这些语言 CPU 能直接理解吗？

答案是：**不能**。CPU 只认识一种语言——**机器码（Machine Code）**，也就是纯粹的二进制指令，比如 `10110000 01100001`。没有任何人愿意直接写机器码编程——那简直是噩梦。

所以人类发明了高级语言（High-Level Language），比如 Python、Java、C++。这些语言接近人类自然语言，写起来方便，读起来也好理解。但问题来了：**CPU 看不懂高级语言，高级语言看不懂 CPU**。

这就需要一个"翻译官"——语言翻译器。

【板书】
```
高级语言 (Source Code)
    ↓ 翻译器 (Translator)
机器码 (Object Code / Machine Code)
    ↓
CPU 执行
```

翻译器有三种：**汇编器（Assembler）**、**编译器（Compiler）**、**解释器（Interpreter）**。我们一个一个来看。

---

## 第一节：汇编器（约 5 分钟）

在高级语言出现之前，程序员用**汇编语言（Assembly Language）**编程。汇编语言是机器码的"人类可读版本"——每条汇编指令基本对应一条机器码指令。

比如：

【板书】
```
汇编语言           机器码
LDA #5         →   00110101 00000101
ADD #3         →   01010010 00000011
STA result     →   01100001 10000100
```

`LDA` 就是 "LoaD the Accumulator"（把值装进累加器），`ADD` 就是加法，`STA` 就是 "STore the Accumulator"（把结果存起来）。这些叫做**助记符（Mnemonics）**——帮助记忆的缩写。

**汇编器（Assembler）**就是把汇编语言翻译成机器码的程序。因为汇编指令和机器码是**一对一**的关系，所以翻译过程相对简单直接。

【提问】既然有了高级语言，为什么还有人用汇编语言？

因为汇编语言可以直接控制硬件，效率极高。在一些对性能要求极端苛刻的场景——比如操作系统内核、嵌入式设备、游戏引擎的核心模块——程序员仍然会用汇编语言来"榨干"CPU 的每一分性能。

---

## 第二节：编译器与解释器（约 15 分钟）

这是本讲最重要的内容。编译器和解释器都是把高级语言翻译成机器码，但翻译方式完全不同。

### 编译器——先翻译，后执行

**编译器（Compiler）**的工作方式就像翻译一本书。翻译者拿到整本英文书，花几个月把它翻译成中文，出版一本中文版。之后读者只需要读中文版就行了，不再需要翻译者。

【板书】
```
编译器的工作流程：
源代码 (.py / .java / .c)
  ↓ 编译器一次性翻译整个文件
目标代码 / 可执行文件 (.exe)
  ↓ 独立运行
CPU 执行（不再需要编译器）
```

### 解释器——边翻译，边执行

**解释器（Interpreter）**的工作方式就像同声传译。演讲者说一句英语，翻译员立刻翻译成中文说出来。如果演讲者再说一遍同样的演讲，翻译员得再翻译一遍——它不会保存上次的翻译结果。

【板书】
```
解释器的工作流程：
源代码 (.py)
  ↓ 解释器逐行翻译并立即执行
  ↓ 每次运行都要重新翻译
CPU 执行（每次都需要解释器）
```

### 关键对比

【板书】
```
┌─────────────┬──────────────────┬──────────────────┐
│  特征        │  Compiler 编译器  │  Interpreter 解释器 │
├─────────────┼──────────────────┼──────────────────┤
│  翻译单位    │  整个程序          │  一行一行          │
│  执行速度    │  快（已编译好）     │  慢（边译边执行）   │
│  开发调试    │  不方便（改了要重编译）│  方便（改了立刻跑） │
│  分发方式    │  发布 .exe 即可    │  必须带源码+解释器  │
│  错误报告    │  一次性列出所有错误  │  遇到第一个错误就停  │
│  源码保护    │  好（只分发 .exe）  │  差（源码暴露）     │
│  典型语言    │  C, C++, Rust     │  Python, JavaScript│
└─────────────┴──────────────────┴──────────────────┘
```

【演示】运行 Cell 5——编译器 vs 解释器的流程可视化。

【演示】运行 Cell 7——编译 vs 解释的速度差异概念演示。

大家注意看这个演示。模拟"编译"时，第一次运行需要等一段时间（编译过程），但之后每次运行都很快（直接执行编译好的代码）。模拟"解释"时，每次运行都需要同样的时间（每次都重新翻译）。

### 什么时候用哪个？

【提问】如果你正在开发一个程序，一边写一边测试，你会选编译器还是解释器？

对，开发阶段用**解释器**更方便——改一行代码就能立刻看到效果。但发布给用户时，用**编译器**更好——运行速度快、源码也受保护。

这就引出了一个聪明的策略——

## 第三节：Java 的混合方式（约 5 分钟）

Java 说："编译器和解释器各有优点，我两个都要！"

【板书】
```
Java 的两步走策略：
Step 1: Java 编译器把 .java 文件编译成 字节码 (.class)
        （不是机器码！是一种中间代码 Bytecode）
Step 2: JVM（Java 虚拟机）在目标机器上解释执行字节码
```

这样做的妙处在于：编译这一步可以检查语法错误、优化代码；而字节码是平台无关的——同一份 .class 文件可以在 Windows、Mac、Linux 上运行，只要那台机器安装了 JVM。这就是 Java 著名的口号：**"Write Once, Run Anywhere"（一次编写，到处运行）**。

【演示】运行 Cell 9——Java 混合方式的可视化。

其实 Python 也有类似的机制——当你运行 .py 文件时，Python 解释器会先把它编译成字节码（.pyc 文件），然后由 Python 虚拟机执行。

【演示】运行 Cell 11——让我们实际看看 Python 字节码长什么样。

大家看，一个简单的 `a + b` 加法，被翻译成了 `LOAD_FAST`、`LOAD_FAST`、`BINARY_ADD`、`RETURN_VALUE` 这样的字节码指令。这些字节码比源代码更接近机器码，但还不是真正的机器码——它们是虚拟机能理解的中间语言。

## 第四节：IDE——开发者的工作坊（约 8 分钟）

现在你知道了翻译器的工作原理。但在实际开发中，程序员不会只用一个文本编辑器和一个翻译器。他们会用一个把所有工具集成在一起的软件——**IDE（Integrated Development Environment，集成开发环境）**。

【提问】你们用过哪些 IDE 或代码编辑器？VS Code？PyCharm？Jupyter Notebook？

IDE 就像一个**全能工作坊**。

【板书】
```
IDE 的核心功能：
1. 代码编辑器 (Code Editor)
   - 语法高亮 (Syntax Highlighting)：不同类型的代码用不同颜色
   - 自动补全 (Auto-complete)：输入几个字母自动提示
   - 自动缩进 (Auto-indent)

2. 编译器/解释器 (Compiler/Interpreter)
   - 一键运行、一键编译

3. 调试器 (Debugger)
   - 断点 (Breakpoint)：让程序在某一行停下来
   - 单步执行 (Step Through)：一行一行地执行
   - 变量监视 (Watch)：实时查看变量值的变化
   - 调用栈 (Call Stack)：查看函数调用层次

4. 错误提示 (Error Detection)
   - 语法错误实时标红
   - 运行时错误的详细信息（行号、错误类型、建议）

5. 版本管理集成 (Version Control)
   - Git 集成
```

让我展开讲讲调试器，因为这是 IDE 最强大的工具。

想象你在破案。程序出了 bug，就像发生了一起"案件"。没有调试器时，你只能靠到处加 `print()` 来猜测——这就像侦探只能挨家挨户敲门问线索。

有了调试器，就像你拥有了时间暂停能力——你可以让程序在任何地方停下来，仔细观察此刻每个变量的值，然后一步一步往前走，看看哪一步出了问题。

【演示】运行 Cell 15——调试概念演示。这段代码模拟了一个有 bug 的函数，展示了如何通过调试思路定位问题。

【演示】运行 Cell 17——语法高亮的原理演示。大家可以看到，语法高亮本质上就是用正则表达式匹配不同类型的代码（关键字、字符串、注释、数字等），然后给它们标上不同颜色。

## 第五节：编译流程概览（约 5 分钟）

最后，我想带大家快速了解一下编译器内部是怎么工作的。虽然考试不会考得很深，但理解整体流程有助于你们理解为什么编译器能发现错误、为什么编译需要时间。

编译器的工作分为多个阶段：

【板书】
```
编译流程：
源代码 → 词法分析 → 语法分析 → 语义分析 → 代码优化 → 代码生成 → 目标代码

词法分析 (Lexical Analysis)：把源代码拆分成一个个"词" (Token)
  例如 "x = 5 + 3" → [变量x] [赋值=] [数字5] [加号+] [数字3]

语法分析 (Syntax Analysis)：检查这些词的排列是否符合语法规则
  例如 "5 + = x" → 语法错误！

语义分析 (Semantic Analysis)：检查含义是否正确
  例如 "hello" + 5 → 字符串和数字不能直接相加！

代码生成 (Code Generation)：生成目标机器码
```

## 第二讲总结（约 2 分钟）

让我们回顾今天的内容：

【板书】
```
语言翻译器
├── 汇编器 (Assembler)：汇编语言 → 机器码（1:1 翻译）

├── 代码编辑器（语法高亮、自动补全）
├── 编译器/解释器（一键运行）
├── 调试器（断点、单步执行、变量监视）
└── 错误提示（实时检查）
```

考试中关于这一节最常见的题目是"Compare compiler and interpreter"——比较编译器和解释器。记住我们讲的翻译书 vs 同声传译的比喻，再结合那张对比表格，就能应对大多数题目了。
