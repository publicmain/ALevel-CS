### 一、开场导入 (5 分钟)

同学们好，今天我们正式进入第 9 章——算法设计与问题解决。在开始之前，我想先问大家一个问题。

【提问】假如有人跟你说："我想做一个手机 App"，你觉得第一步应该做什么？是打开电脑开始写代码吗？

……对，不是。就像你想盖一栋房子，你不会拿起砖头就开始砌墙。你得先想清楚：这个房子要几层？住几个人？预算多少？然后你要画设计图，分好工种，最后才动手施工。

写程序也是一样的。真正优秀的程序员，70% 的时间花在**思考和设计**上，只有 30% 的时间在写代码。今天我们要学的，就是这个"思考"的方法论——**计算思维 (Computational Thinking)**。

计算思维不等于编程。编程只是把你思考的结果翻译成计算机能懂的语言。计算思维才是真正核心的能力。Google、Amazon 这些大厂面试，考的不是你会不会某种语言的语法，而是你能不能系统地分析和解决问题。

计算思维有四大核心技能：**抽象 (Abstraction)**、**分解 (Decomposition)**、**模式识别 (Pattern Recognition)**，以及**算法设计 (Algorithm Design)**。今天我们重点讲前三个。

### 二、抽象 (Abstraction) (15 分钟)

#### 2.1 什么是抽象？

【板书】抽象 = 隐藏不必要的细节，只保留解决问题所需的关键信息

这个概念听起来很"抽象"对吧？但其实你们每天都在用它。

你们都用手机对吧？你按一下微信图标就能发消息。你需要知道 CPU 是怎么处理你的触摸信号的吗？需要知道网络数据包是怎么通过路由器传输的吗？不需要。手机的界面帮你**隐藏了所有底层的复杂细节**，只留给你最关键的东西——按钮和图标。这就是抽象。

再比如开车。你关注的是方向盘、油门、刹车，你不需要知道发动机里面汽油是怎么燃烧的。汽车的设计者帮你做了抽象，把复杂的机械原理简化成了几个你能操作的控件。

#### 2.2 经典案例：伦敦地铁图

【演示】运行 Notebook 中伦敦地铁图的对比展示

这是计算机科学中讲抽象最经典的例子。1933 年，一个叫 Harry Beck 的工程师设计了伦敦地铁图。我们来对比一下真实的地理地图和他设计的地铁图。

【板书】

| 方面 | 真实地理 | 地铁图（抽象后） |
|------|---------|-----------------|
| 站点距离 | 精确但复杂 | 等间距，简洁 |
| 线路走向 | 弯弯曲曲 | 只用水平/垂直/45° 线 |
| 地面建筑 | 密密麻麻 | 完全不显示 |

他去掉了什么？去掉了真实的距离、街道、建筑物。他保留了什么？保留了站点名称、线路颜色、换乘关系。

为什么这样做？因为坐地铁的人只关心"我在哪一站上车、坐哪条线、在哪一站换乘、在哪一站下车"。至于两站之间到底隔了 500 米还是 2 公里，坐地铁的人根本不在意。

这就是抽象的精髓：**根据你要解决的问题，决定保留什么、去掉什么**。

【提问】如果你要设计一个学校的课程表系统，你会对"学生"做怎样的抽象？保留哪些信息，去掉哪些？

……对，你可能保留姓名、学号、年级、选课信息，但去掉身高、体重、爱好这些跟排课无关的信息。如果你做的是体育选拔系统，那身高体重可能就要保留了。**抽象的关键在于"跟问题相关"**。

#### 2.3 计算机科学中的抽象层次

抽象在计算机科学中无处不在。我们之前学过的内容里就充满了抽象的例子：

- **高级编程语言**是对机器码的抽象——你写 `print("Hello")` 而不是 `10110000 01001000...`
- **操作系统**是对硬件的抽象——你用"文件"和"文件夹"而不是直接操作磁盘扇区
- **TCP/IP 网络模型**是分层抽象——每一层只关心自己该做的事

### 三、分解 (Decomposition) (15 分钟)

#### 3.1 什么是分解？

【板书】分解 = 把一个大问题拆成多个小问题，分别解决

我们继续用盖房子来打比方。假设你要盖一栋房子，这个任务太大了对不对？你不可能一个人同时干所有事。你会怎么做？

你会把它拆开：首先是**地基工程**，然后是**主体结构**，接着是**水电安装**、**室内装修**，最后是**验收**。每个部分可以交给不同的专业团队去做，最后组装在一起。

程序设计完全一样。如果有人让你"做一个学校管理系统"，你不会傻傻地写一个超级长的程序。你会把它拆成：**学生管理模块**、**课程管理模块**、**成绩管理模块**、**教师管理模块**，然后一个一个解决。

#### 3.2 分解的原则

【板书】好的分解应该满足：
1. 每个子问题可以独立解决
2. 子问题比原问题简单得多
3. 所有子问题组合起来等于原问题

【提问】假如我要你们设计一个自动售货机的程序，你们会怎么分解这个问题？

……很好。我们可以分成这样几个子问题：
1. **显示商品列表**——让用户看到有什么可买
2. **接受用户选择**——用户按按钮选商品
3. **处理付款**——投币或刷卡
4. **找零**——如果多付了要找钱
5. **出货**——把商品送到出口

每一个子问题都比"做一个自动售货机程序"简单得多。这就是分解的威力。

#### 3.3 分解的实践

【演示】运行 Notebook 中分解示例的代码

分解不仅仅是一种思维方式，它还直接对应到编程中的概念。我们后面在第 11 章学结构化编程时，会用到**过程 (Procedure)** 和**函数 (Function)**，每一个过程或函数就是解决一个子问题。分解得好，你的代码就清晰、好维护；分解得差，你的代码就乱成一团。

### 四、模式识别 (Pattern Recognition) (10 分钟)

#### 4.1 什么是模式识别？

【板书】模式识别 = 发现不同问题之间的相似性，复用已有的解决方案

你们有没有发现，学数学的时候，很多看起来不一样的题，解法其实是一样的？比如"鸡兔同笼"和"票价计算"，本质上都是解二元一次方程组。当你发现了这个"模式"，你就不用每道题都从头想解法了。

编程也是一样。很多看似不同的问题，底层的模式是相同的：
- 在学生列表中找最高分 → 在商品列表中找最低价 → 在温度记录中找最大值 → **都是"遍历找极值"的模式**
- 用户登录验证 → ATM 密码验证 → 手机解锁 → **都是"输入-比对-反馈"的模式**

#### 4.2 模式识别在实际开发中的应用

当你积累了足够多的编程经验后，你会发现大部分问题都可以归类到有限的几种"模式"中：
- 搜索模式（在数据中找东西）
- 排序模式（把数据按顺序排列）
- 累加模式（把数据累加/统计）
- 过滤模式（按条件筛选数据）

我们下节课以及第 10 章会详细学习这些具体的算法模式。

### 五、课堂小结 (5 分钟)

【板书】

| 计算思维技能 | 核心思想 | 类比 |
|-------------|---------|------|
| 抽象 | 去掉无关细节，保留关键信息 | 地铁图 |
| 分解 | 大问题拆成小问题 | 盖房子分工 |
| 模式识别 | 发现相似性，复用方案 | 数学解题套路 |

这三个技能不是孤立的，而是配合使用的。面对一个问题，你先用**抽象**搞清楚关键信息是什么，再用**分解**把问题拆小，然后用**模式识别**看看有没有已知的解决方案可以借鉴。最后，你把这些思考的结果写成一步步的指令，这就是**算法**——我们下节课的主题。

### 一、什么是算法？(5 分钟)

同学们，上节课我们学了计算思维的三大技能。今天我们来学计算思维的最终产出——**算法 (Algorithm)**。

【板书】算法 = 解决特定问题的、有限的、明确的步骤序列

注意这几个关键词：
- **有限的 (Finite)**：算法必须会结束，不能无限运行下去
- **明确的 (Unambiguous)**：每一步都必须清楚明确，不能有歧义
- **有输入输出**：接受输入，产生输出

其实你们生活中天天在用算法。泡方便面的步骤就是一个算法：1. 打开包装 → 2. 放入调料 → 3. 加开水 → 4. 盖上盖子等 3 分钟 → 5. 开吃。有限步骤，每步明确，有输入（面和水），有输出（一碗泡面）。

但是算法得写下来给别人看，甚至给计算机看。所以我们需要一种**表示方法**。今天我们学三种：**结构化英语 (Structured English)**、**流程图 (Flowchart)**、**伪代码 (Pseudocode)**。

---

### 二、标识符表 (Identifier Table) (5 分钟)

在写算法之前，我们需要先想清楚：这个算法需要用到哪些数据？每个数据叫什么名字？是什么类型的？

【板书】标识符表 (Identifier Table)

| 标识符 (Identifier) | 数据类型 (Data Type) | 描述 (Description) |
|---------------------|---------------------|--------------------|
| Name | STRING | 学生姓名 |
| Age | INTEGER | 学生年龄 |
| Score | REAL | 考试成绩 |
| Pass | BOOLEAN | 是否及格 |

为什么要做这个表？因为：
1. 让你在写算法之前想清楚需要什么数据
2. 方便别人阅读和理解你的算法
3. 考试的时候经常要你写这个表！

命名规则：
- 使用有意义的名字：`StudentName` 而不是 `x`
- 首字母大写（Cambridge 伪代码的惯例）
- 不能用保留字（如 IF, WHILE, FOR 等）

### 三、算法的三种表示 (25 分钟)

好，现在我们来看一个具体的例子。假设我们要解决这样一个问题：

> 输入一个学生的分数（0-100），如果 >= 50 分就输出 "Pass"，否则输出 "Fail"。

这个问题很简单对吧？但我们用它来展示同一个算法的三种表示方式。

#### 3.1 结构化英语 (Structured English)

结构化英语就是用比较规范的英语来描述步骤，介于自然语言和编程语言之间。

【板书】

```
1. Get the student's score
2. IF the score is greater than or equal to 50
       THEN display "Pass"
       OTHERWISE display "Fail"
3. End
```

它的好处是：非程序员也能看懂。客户、老师、项目经理都能理解。它的缺点是：不够精确，不同人理解可能不一样。

#### 3.2 流程图 (Flowchart)

流程图用图形符号来表示算法。

【板书】流程图的基本符号：

| 符号 | 形状 | 用途 |
|------|------|------|
| 起止框 | 圆角矩形 | 算法的开始和结束 |
| 处理框 | 矩形 | 执行操作（赋值、计算） |
| 判断框 | 菱形 | 条件判断（Yes/No 分支） |
| 输入/输出框 | 平行四边形 | 输入数据或输出结果 |
| 流程线 | 箭头 | 连接各步骤 |

现在我来画这个判断及格的流程图。

【板书】（画流程图）

```
    [开始]
       ↓
   /输入 Score/
       ↓
   <Score >= 50?> ── Yes ──→ /输出 "Pass"/
       │                          ↓
       No                      [结束]
       ↓
   /输出 "Fail"/
       ↓
    [结束]
```

【演示】运行 Notebook 中流程图的可视化

流程图的好处是**直观**，一眼就能看到程序的流向。特别是有分支和循环的时候，流程图能让逻辑非常清楚。但缺点是：复杂算法画起来会很大很乱。

#### 3.3 伪代码 (Pseudocode)

伪代码是最接近真实编程语言的表示方式，但又不是任何一种具体的语言。Cambridge 考试有自己的伪代码规范，我们必须按照这个规范来写。

【板书】同一个算法的伪代码：

```
DECLARE Score : INTEGER
OUTPUT "Enter score: "
INPUT Score
IF Score >= 50
  THEN
    OUTPUT "Pass"
  ELSE
    OUTPUT "Fail"
ENDIF
```

注意几个要点：
- **DECLARE** 声明变量及其类型
- **INPUT** 读取用户输入
- **OUTPUT** 输出显示
- **IF...THEN...ELSE...ENDIF** 条件判断
- 赋值用 **←** 符号（如 `Score ← 0`）

【提问】大家对比一下这三种表示方式，你觉得各有什么优缺点？

……对，结构化英语最好懂但最不精确；流程图最直观但复杂时画不下；伪代码最精确，最接近编程，但需要学语法规则。考试中伪代码是最重要的。

#### 3.4 伪代码的三大结构

任何算法——不管多复杂——都只由三种基本结构组成。这是一个非常重要的结论。

【板书】

**1. 顺序 (Sequence)**
```
语句 1
语句 2
语句 3
```
按顺序一条一条执行，没什么好说的。

**2. 选择 (Selection)**
```
IF condition
  THEN
    statements
  ELSE
    statements
ENDIF
```
或者多条件选择：
```
CASE OF variable
  value1 : statement1
  value2 : statement2
  OTHERWISE : statement3
ENDCASE
```

**3. 迭代/循环 (Iteration)**
```
FOR i ← 1 TO n
  statements
NEXT i

WHILE condition DO
  statements
ENDWHILE

REPEAT
  statements
UNTIL condition
```

这三个结构就像乐高积木里的基本砖块。所有复杂的程序，本质上就是这三种结构的组合和嵌套。

### 四、逐步细化 (Stepwise Refinement) (5 分钟)

有了分解和伪代码的知识，我们可以把它们结合起来。

【板书】逐步细化 = 先写粗略的步骤，然后逐步将每一步细化成更具体的伪代码

举个例子，计算一个班级的平均分：

**第一层（粗略）**：
```
1. 输入所有学生的分数
2. 计算总分
3. 计算平均分
4. 输出结果
```

**第二层（细化）**：
```
1. 输入所有学生的分数
   1.1 设置 Total ← 0, Count ← 0
   1.2 重复：输入一个分数，加到 Total，Count 加 1
   1.3 直到所有分数输入完毕
2. 计算平均分
   2.1 Average ← Total / Count
3. 输出 Average
```

**第三层（伪代码）**：
```
DECLARE Total : REAL
DECLARE Count : INTEGER
DECLARE Score : REAL
DECLARE Average : REAL

Total ← 0
Count ← 0
REPEAT
  OUTPUT "Enter score (-1 to stop): "
  INPUT Score
  IF Score <> -1
    THEN
      Total ← Total + Score
      Count ← Count + 1
  ENDIF
UNTIL Score = -1
Average ← Total / Count
OUTPUT "Average = ", Average
```

看到了吗？从模糊到清晰，一层一层细化。这就是逐步细化的过程。

### 五、逻辑语句和综合示例 (8 分钟)

#### 5.1 逻辑语句

在写条件判断时，我们经常需要用到逻辑运算符。

【板书】

| 运算符 | 含义 | 示例 |
|--------|------|------|
| AND | 两个条件都为真 | `Age >= 18 AND HasID = TRUE` |
| OR | 至少一个为真 | `Score >= 90 OR HasBonus = TRUE` |
| NOT | 取反 | `NOT GameOver` |

复杂条件要用括号确保优先级正确：
```
IF (Age >= 18 AND Age <= 65) AND (HasTicket = TRUE)
  THEN
    OUTPUT "Allowed entry"
ENDIF
```

#### 5.2 综合示例

现在我们来做一个完整的例子。写一个算法：输入 10 个学生的成绩，统计及格人数（>= 50 分），并输出及格率。

我们从三种形式都展示一遍。

**结构化英语**：
```
1. Set pass count to zero
2. FOR each of 10 students
     a. Get the student's score
     b. IF score >= 50 THEN add 1 to pass count
3. Calculate pass rate as pass count / 10 * 100
4. Display the pass rate
```

**流程图**：

【板书】（在白板上画包含 FOR 循环的流程图）

**伪代码**：
```
DECLARE Score : INTEGER
DECLARE PassCount : INTEGER
DECLARE PassRate : REAL

PassCount ← 0
FOR i ← 1 TO 10
  OUTPUT "Enter score for student ", i, ": "
  INPUT Score
  IF Score >= 50
    THEN
      PassCount ← PassCount + 1
  ENDIF
NEXT i
PassRate ← (PassCount / 10) * 100
OUTPUT "Pass rate: ", PassRate, "%"
```

【演示】运行 Notebook 中综合示例的代码

【提问】如果我要改成"不固定 10 个学生，而是输入 -1 表示结束"，应该把 FOR 循环改成什么循环？

……对了，改成 REPEAT...UNTIL 或 WHILE 循环，因为我们不知道会输入多少个成绩。
