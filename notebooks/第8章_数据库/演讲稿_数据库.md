## 开场：一个没有数据库的世界（约 8 分钟）

同学们好！今天我们开始学习第八章——数据库。这一章非常重要，不仅考试分值高，而且是你们未来做任何软件开发都绑定不了的核心知识。

我先带大家想象一个场景。

【提问】假设你是一所学校的校长，需要管理所有学生的信息。但是，你没有数据库。你会怎么做？

你可能会说："用 Excel 表格呗。"好，那我们想象一下更原始的情况——用纸质文件和文件柜。

学术部门有一个文件柜，里面存着每个学生的姓名、年龄、班级、成绩。财务部门也有一个文件柜，存着每个学生的姓名、年龄、缴费记录。图书馆也有一个，存着姓名、年龄、借书记录。

你看到问题了吗？**同一个学生的姓名和年龄被抄了三份**——分别在三个文件柜里。

这就是数据库出现之前的**文件处理方法（File-Based Approach）**。在计算机上也一样——每个部门有自己独立的数据文件，互不相通。

### 文件处理方法的五大问题

【板书】
```
1. 数据冗余 (Data Redundancy)
   同样的数据被存储多次 → 浪费空间

2. 数据不一致 (Data Inconsistency)
   学生改了名字，学术部门改了，但财务部门忘了改
   → 同一个学生在不同文件里的名字不一样

3. 数据孤岛 (Data Isolation)
   每个部门的数据格式不同，无法轻松合并查询
   → "哪些成绩优秀的学生还欠学费？"这种跨部门查询几乎不可能

4. 数据依赖 (Program-Data Dependence)
   文件格式写死在程序里 → 文件结构一改，所有程序都要跟着改

5. 缺乏安全控制
   文件要么全给访问权限，要么全不给
   → 无法做到"允许看成绩，但不允许看工资"
```

【演示】运行 Cell 3——五大问题的可视化图表。

【演示】运行 Cell 5——用 Python 模拟文件处理方法的问题。大家看代码里，三个部门各自维护一份学生数据，当学生改名时，只有一个部门更新了，其他两个还是旧名字——这就是数据不一致。

为了解决这些问题，E.F. Codd 在 1970 年提出了**关系数据库模型（Relational Database Model）**——这是计算机科学史上最重要的发明之一。

## 第一节：关系数据库术语（约 10 分钟）

关系数据库用**表（Table）**来组织数据。但在考试中，你必须使用精确的术语。

【板书】
```
数据库术语大全（考试必背）：

表 (Table) = 关系 (Relation) = 实体 (Entity)
  存储一类事物的所有数据
  例子：STUDENTS 表存储所有学生信息

行 (Row) = 记录 (Record) = 元组 (Tuple)
  表中的一行，代表一个具体的实例
  例子：一行 = 一个学生的全部信息

列 (Column) = 字段 (Field) = 属性 (Attribute)
  表中的一列，代表一种信息
  例子：Name 列 = 所有学生的姓名

主键 (Primary Key)
  唯一标识每一行的字段（不能重复，不能为空）
  例子：StudentID（学号不会重复）

外键 (Foreign Key)
  一个表中引用另一个表主键的字段
  作用：建立表与表之间的"关系"
  例子：ENROLLMENTS 表中的 StudentID 引用 STUDENTS 表的 StudentID

候选键 (Candidate Key)
  所有可以作为主键的字段（可能有多个）
  例子：学号和身份证号都能唯一标识学生，它们都是候选键
  最终选一个做主键，其他的叫"备选键 (Alternate Key)"
```

【演示】运行 Cell 8——用 Python 创建示例表并展示所有术语。

### 关系类型

表与表之间通过外键建立关系。关系有三种：

【板书】
```
三种关系类型：

1. 一对一 (1:1) —— One-to-One
   一个丈夫 ↔ 一个妻子
   一个国家 ↔ 一个首都

2. 一对多 (1:M) —— One-to-Many（最常见！）
   一个班级 → 多个学生
   一个老师 → 多门课程

3. 多对多 (M:N) —— Many-to-Many
   一个学生 ↔ 多门课程
   一门课程 ↔ 多个学生

注意：关系数据库不能直接实现 M:N 关系！
解决方法：拆成两个 1:M，中间加一个"连接表 (Junction Table)"

STUDENTS ←1:M→ ENROLLMENTS ←M:1→ COURSES
```

## 第三节：规范化——数据库设计的核心技能（约 15 分钟）

规范化（Normalization）是将数据库设计得更高效、更可靠的过程。它是考试的**重点中的重点**。

【提问】为什么要规范化？直接把所有数据放一张大表里不行吗？

当然可以——但会出现很多问题。比如一张大表里有学生信息、课程信息、老师信息：

【板书】
```
一张大表（未规范化 UNF）：
┌──────────┬──────┬────────┬────────┬────────┬────────────┐
│StudentID │Name  │CourseID│Course  │Teacher │TeacherPhone│
├──────────┼──────┼────────┼────────┼────────┼────────────┤
│S001      │Alice │CS101   │Comp Sci│Mr.Wang │1234567     │
│S001      │Alice │MA101   │Math    │Ms.Li   │2345678     │
│S002      │Bob   │CS101   │Comp Sci│Mr.Wang │1234567     │
│S002      │Bob   │EN101   │English │Mr.Chen │3456789     │
└──────────┴──────┴────────┴────────┴────────┴────────────┘

问题：
- Alice 的名字出现了2次（冗余）
- Mr.Wang 的电话出现了2次（冗余）
- 如果 Mr.Wang 换电话，要改多行（更新异常）
- 如果删除 Alice 的所有选课，Alice 的信息也丢了（删除异常）
- 如果一门新课还没有学生选，就无法录入（插入异常）
```

规范化就是通过拆表来解决这些问题。一共三个范式：

### 第一范式（1NF）

【板书】
```
1NF 规则：
1. 每个字段只包含原子值（atomic value）——不能是列表或集合
2. 没有重复组（repeating groups）
3. 有主键

违反 1NF 的例子：
┌──────────┬──────┬──────────────────┐
│StudentID │Name  │Courses           │
├──────────┼──────┼──────────────────┤
│S001      │Alice │CS101, MA101      │  ← Courses 字段包含多个值！
│S002      │Bob   │CS101, EN101      │
└──────────┴──────┴──────────────────┘

转换为 1NF：把多值字段拆开，每行只有一个值
┌──────────┬──────┬────────┐
│StudentID │Name  │CourseID│
├──────────┼──────┼────────┤
│S001      │Alice │CS101   │
│S001      │Alice │MA101   │
│S002      │Bob   │CS101   │
│S002      │Bob   │EN101   │
└──────────┴──────┴────────┘
主键 = (StudentID, CourseID) —— 复合主键
```

### 第二范式（2NF）

【板书】
```
2NF 规则：
1. 已经是 1NF
2. 没有部分依赖（Partial Dependency）
   → 非键字段必须依赖于整个主键，不能只依赖主键的一部分

违反 2NF 的例子：
主键 = (StudentID, CourseID)
Name 只依赖于 StudentID（不需要 CourseID）→ 部分依赖！

解决：把只依赖部分主键的字段拆到新表

拆分后：
STUDENTS 表：StudentID (PK), Name
ENROLLMENTS 表：StudentID (FK), CourseID (FK)
```

### 第三范式（3NF）

【板书】
```
3NF 规则：
1. 已经是 2NF
2. 没有传递依赖（Transitive Dependency）
   → 非键字段不能依赖于另一个非键字段

违反 3NF 的例子：
COURSES 表：CourseID (PK), CourseName, TeacherID, TeacherPhone
TeacherPhone 依赖于 TeacherID，而 TeacherID 不是主键
→ 传递依赖：CourseID → TeacherID → TeacherPhone

解决：拆出 TEACHERS 表
COURSES 表：CourseID (PK), CourseName, TeacherID (FK)
TEACHERS 表：TeacherID (PK), TeacherPhone
```

【演示】运行 Cell 14、Cell 15、Cell 16——逐步展示规范化过程。

【演示】运行 Cell 19——互动规范化练习。

规范化的核心口诀：

【板书】
```
1NF：消除重复组，保证原子值
2NF：消除部分依赖（非键字段完全依赖主键）
3NF：消除传递依赖（非键字段不依赖非键字段）
```

## 第一节：DBMS 的核心功能（约 20 分钟）

### 1. 数据字典（Data Dictionary）

数据字典是**关于数据的数据（metadata）**——它记录数据库本身的结构信息。

【板书】
```
数据字典记录了：
- 所有表的名称
- 每个表有哪些字段（列名、数据类型、约束）
- 主键和外键信息
- 索引信息
- 用户权限
- 表之间的关系

类比：如果数据库是一本书，数据字典就是"目录+前言"
——告诉你这本书有哪些章节、每章多少页、用什么语言写的
```

【演示】运行 Cell 10——用 sqlite_master 查看 SQLite 的数据字典。大家看，我们执行一个查询就能看到数据库里有哪些表、每个表是怎么创建的。这就是数据字典的作用——它是 DBMS 自动维护的。

### 2. 数据完整性（Data Integrity）

DBMS 通过**约束（Constraints）**来保证数据的正确性。

【板书】
```
五种主要约束：
┌──────────────┬──────────────────┬─────────────────────┐
│ 约束类型      │ 作用              │ 例子                 │
├──────────────┼──────────────────┼─────────────────────┤
│ NOT NULL     │ 字段不能为空       │ 学生姓名不能为空      │
│ UNIQUE       │ 字段值必须唯一     │ 邮箱不能重复          │
│ PRIMARY KEY  │ 唯一 + 非空       │ StudentID            │
│ FOREIGN KEY  │ 引用必须存在       │ ClassID 必须在班级表中│
│ CHECK        │ 自定义条件         │ Age BETWEEN 5 AND 100│
└──────────────┴──────────────────┴─────────────────────┘
```

【演示】运行 Cell 10 的完整性验证部分——看看当我们试图插入重复的主键或空的姓名时，DBMS 如何自动拒绝。

这就是 DBMS 的强大之处——你不需要在每个应用程序里写检查代码，DBMS 会在数据层面自动保护。

### 3. 安全与访问控制

DBMS 提供精细的权限控制：

【板书】
```
权限层次：
DBA（数据库管理员）→ 最高权限，可以做任何事
开发者 → 可以创建/修改表结构
普通用户 → 只能查询指定的表
只读用户 → 只能 SELECT，不能修改

SQL 权限语句：
GRANT SELECT ON Students TO teacher_role;     -- 老师只能查看学生表
GRANT SELECT, INSERT, UPDATE ON Grades TO teacher_role;  -- 老师可以录入成绩
GRANT ALL ON ALL TABLES TO admin_role;        -- 管理员全权限
```

### 4. 并发访问控制——最难理解的部分

当多个用户同时操作数据库时，可能出现严重问题。

【板书】
```
丢失更新问题 (Lost Update Problem)：

账户余额 = ¥1000

用户A（取款 ¥500）：          用户B（转账 ¥300）：
1. 读取余额 = ¥1000           1. 读取余额 = ¥1000
2. 计算: 1000-500 = ¥500      2. 计算: 1000-300 = ¥700
3. 写入余额 = ¥500            3. 写入余额 = ¥700

最终余额 = ¥700（或 ¥500），但正确应该是 ¥200！
用户A 的操作被用户B "覆盖" 了。
```

【提问】这个问题怎么解决？

DBMS 使用**锁（Locking）**机制：

【板书】
```
锁的类型：
- 读锁 (Shared Lock)：多人可以同时读，但不能改
- 写锁 (Exclusive Lock)：一个人在改时，其他人不能读也不能改

事务 (Transaction)：一组操作，要么全部成功，要么全部失败
→ ACID 原则：
  A - Atomicity（原子性）：事务不可分割
  C - Consistency（一致性）：事务前后数据一致
  I - Isolation（隔离性）：事务之间互不干扰
  D - Durability（持久性）：完成的事务永久保存
```

【演示】运行 Cell 12——银行转账的事务演示。大家看，从 Alice 的账户扣 200，给 Bob 的账户加 200——这两个操作要么同时成功，要么同时失败。如果扣了 Alice 的钱但 Bob 那边出错了，整个事务回滚（Rollback），Alice 的钱恢复原样。

### 5. 备份与恢复

数据库需要定期备份，以防硬件故障、软件错误、人为失误或自然灾害。

### 6. 查询处理器

查询处理器是 DBMS 的"大脑"：

【板书】
```
查询处理三步走：
1. 解析 (Parsing)：检查 SQL 语法是否正确
2. 优化 (Optimization)：找到执行查询的最佳方式
3. 执行 (Execution)：访问数据并返回结果

为什么需要优化？
查询"所有北京的17岁学生"有两种方式：
方式A：先找北京学生(1000人)，再筛17岁 → 扫描1000条
方式B：先找17岁学生(50人)，再筛北京 → 扫描50条
查询处理器自动选最快的！
```

## 第二节：DBMS vs 文件处理（约 5 分钟）

这是考试经常出的对比题。

【板书】
```
┌──────────┬───────────────┬─────────────┐
│ 方面      │ 文件处理方法   │ DBMS        │
├──────────┼───────────────┼─────────────┤
│ 数据冗余  │ 大量重复       │ 最小化       │
│ 数据一致性│ 难以保证       │ 自动保证     │
│ 数据共享  │ 困难           │ 多用户并发   │
│ 数据完整性│ 无自动检查     │ 约束自动检查  │
│ 安全性    │ 基本没有       │ 精细权限控制  │
│ 并发控制  │ 无             │ 锁+事务      │
│ 数据独立性│ 无             │ 程序与数据分离│
└──────────┴───────────────┴─────────────┘
```

**数据独立性（Data Independence）**是 DBMS 最重要的优势之一。它有两个层面：
- **逻辑数据独立性**：改变表结构不影响应用程序
- **物理数据独立性**：改变存储方式不影响逻辑结构

类比：你换了一个更大的书架（物理变化），书的分类方式（逻辑结构）不变，你的借书习惯（应用程序）也不需要改变。

## 开场：SQL——数据库的"普通话"（约 3 分钟）

同学们好！前两节课我们学了数据库的概念和 DBMS 的功能。今天是重头戏——**SQL（Structured Query Language，结构化查询语言）**。

SQL 是和数据库"交流"的语言。不管你用的是 MySQL、PostgreSQL、Oracle 还是 SQLite，SQL 的核心语法都是一样的。它就像数据库世界的"普通话"。

SQL 分为两大类：

【板书】
```
DDL (Data Definition Language) — 数据定义语言
→ 定义数据库的结构（创建表、修改表、删除表）
→ CREATE, ALTER, DROP

DML (Data Manipulation Language) — 数据操作语言
→ 操作表中的数据（增删改查）
→ SELECT, INSERT, UPDATE, DELETE
```

今天我们会逐一学习这些命令。先让我们建立一个练习用的数据库。

### 聚合函数

【板书】
```sql
SELECT COUNT(*) FROM Students;            -- 计数
SELECT AVG(Score) FROM Enrollments;       -- 平均值
SELECT SUM(Credits) FROM Courses;         -- 总和
SELECT MAX(Score) FROM Enrollments;       -- 最大值

-- GROUP BY：按组聚合

```

【提问】WHERE 和 HAVING 有什么区别？

## 第五节：JOIN——最强大的功能（约 10 分钟）

JOIN 是 SQL 最核心、也是考试最常考的功能。它让你把多个相关的表**组合**在一起查询。

### 为什么需要 JOIN？

因为我们做了规范化——把数据拆到了多个表里。但查询的时候，我们经常需要"跨表"的信息。比如"每个学生选了什么课，成绩多少"——这个信息分散在 Students、Enrollments、Courses 三个表里。

【板书】
```sql
-- 基本 JOIN 语法（INNER JOIN）
SELECT Students.FirstName, Courses.CourseName, Enrollments.Score
FROM Students
JOIN Enrollments ON Students.StudentID = Enrollments.StudentID
JOIN Courses ON Enrollments.CourseID = Courses.CourseID;

-- JOIN 的工作原理：
-- 1. 找到 Students 和 Enrollments 中 StudentID 相同的行
-- 2. 找到 Enrollments 和 Courses 中 CourseID 相同的行
-- 3. 把匹配的行组合成一个大表
```

【演示】运行 Cell 27——JOIN 查询示例。

大家注意看结果——每一行都包含了学生名字、课程名字和成绩。这些信息原本分散在三个表里，JOIN 把它们组合在了一起。

### 更复杂的 JOIN

JOIN 可以和 WHERE、ORDER BY、聚合函数等组合使用：

【板书】
```sql
-- 每门课的平均分（包含课程名称）
SELECT Courses.CourseName, AVG(Enrollments.Score) AS AvgScore
FROM Courses
JOIN Enrollments ON Courses.CourseID = Enrollments.CourseID
GROUP BY Courses.CourseName
ORDER BY AvgScore DESC;
```

### 从简单到复杂的查询构建步骤

【板书】
```
构建复杂查询的步骤：
1. 确定需要哪些列（SELECT 什么）
2. 确定需要哪些表（FROM 什么）
3. 确定表之间的关系（JOIN 条件）
4. 确定过滤条件（WHERE）
5. 确定分组（GROUP BY）
6. 确定分组过滤（HAVING）
7. 确定排序（ORDER BY）
```

【演示】运行 Cell 31——一步步构建复杂查询的演示。从一个简单的 SELECT 开始，逐步加入 JOIN、WHERE、GROUP BY、HAVING、ORDER BY。

【提问】让我们一起做一道题：找出"平均分在 80 分以上的 Year 12 学生"。

第一步：需要什么列？学生名字、平均分
第二步：需要什么表？Students、Enrollments
第三步：JOIN 条件？Students.StudentID = Enrollments.StudentID
第四步：WHERE 条件？YearGroup = 12
第五步：GROUP BY？Students.StudentID
第六步：HAVING？AVG(Score) > 80

【板书】
```sql
SELECT Students.FirstName, Students.LastName, AVG(Enrollments.Score) AS AvgScore
FROM Students
JOIN Enrollments ON Students.StudentID = Enrollments.StudentID
WHERE Students.YearGroup = 12
GROUP BY Students.StudentID, Students.FirstName, Students.LastName
HAVING AVG(Enrollments.Score) > 80
ORDER BY AvgScore DESC;
```

## 第三讲总结（约 2 分钟）

让我们回顾 SQL 的所有命令：

【板书】
```
SQL 命令速查：

DDL（定义结构）：
  CREATE TABLE ... ← 创建表
  ALTER TABLE ...  ← 修改表
  DROP TABLE ...   ← 删除表

DML（操作数据）：
  INSERT INTO ... VALUES ...   ← 插入
  SELECT ... FROM ... WHERE ... ← 查询
  UPDATE ... SET ... WHERE ...  ← 更新
  DELETE FROM ... WHERE ...     ← 删除

高级查询：
  ORDER BY  ← 排序
  LIKE      ← 模式匹配（%, _）
  聚合函数   ← COUNT, AVG, SUM, MAX, MIN
  GROUP BY  ← 分组
  HAVING    ← 分组后过滤
  JOIN ... ON ← 连接多表

关键记忆：
- WHERE 在 GROUP BY 之前过滤行
- HAVING 在 GROUP BY 之后过滤组
- UPDATE/DELETE 不加 WHERE = 灾难
- JOIN 用 ON 指定连接条件
```
